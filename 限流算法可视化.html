限流算法：固定窗口 vs 滑动窗口

限流是保护系统免受流量冲击的关键技术，通过控制单位时间内的请求数量来确保系统稳定。 以下展示两种常见的限流算法：固定窗口限流和滑动窗口限流的工作原理与差异。

固定窗口限流

将时间划分为固定长度的区间（如1秒），每个区间独立计数，当请求数超过阈值时触发限流。 实现简单但可能在窗口切换时出现流量突刺。

滑动窗口限流

时间窗口随时间动态滑动，通过统计窗口内的请求总数判断是否限流。 流量控制更平滑，避免了固定窗口的边界突刺问题，但实现相对复杂。

核心代码示例

固定窗口限流 (Java + Redis) 

/**
 * 固定窗口限流工具类
 */
@Component
public class RedisFixedWindowLimiter {

    private final RedisTemplate redisTemplate;

    public RedisFixedWindowLimiter(RedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    /**
     * 固定窗口限流判断
     * @param keyPrefix 限流key前缀
     * @param windowSize 窗口大小(秒)
     * @param limit 窗口内限流阈值
     * @return true: 允许请求; false: 限流
     */
    public boolean isAllowed(String keyPrefix, int windowSize, int limit) {
        // 生成限流key，基于当前时间窗口
        long currentWindow = System.currentTimeMillis() / (windowSize * 1000);
        String limitKey = keyPrefix + ":fixed:limiter:" + currentWindow;
        
        // 自增计数
        Long count = redisTemplate.opsForValue().increment(limitKey);
        
        // 第一次设置过期时间
        if (count != null && count == 1) {
            redisTemplate.expire(limitKey, windowSize, TimeUnit.SECONDS);
        }
        
        // 判断是否超过阈值
        return count != null && count <= limit;
    }
}

核心逻辑：将时间划分为固定窗口，每个窗口独立计数，超过阈值则限流。实现简单但存在窗口边界流量突刺问题。

滑动窗口限流 (Java + Redis) 

/**
 * 滑动窗口限流工具类
 */
@Component
public class RedisSlideWindowLimiter {

    private final RedisTemplate redisTemplate;
    
    // Lua脚本保证操作原子性
    private static final String LUA_SCRIPT = "local key = KEYS[1]\n" +
            "local windowSize = tonumber(ARGV[1])\n" +
            "local limit = tonumber(ARGV[2])\n" +
            "local requestId = ARGV[3]\n" +
            "local currentTimestamp = tonumber(ARGV[4])\n" +
            "\n" +
            "-- 清理时间窗口外的请求\n" +
            "local expireTimestamp = currentTimestamp - windowSize\n" +
            "redis.call('ZREMRANGEBYSCORE', key, 0, expireTimestamp)\n" +
            "\n" +
            "-- 添加当前请求\n" +
            "redis.call('ZADD', key, currentTimestamp, requestId)\n" +
            "\n" +
            "-- 设置过期时间\n" +
            "redis.call('EXPIRE', key, windowSize + 1)\n" +
            "\n" +
            "-- 统计当前窗口内的请求数\n" +
            "local count = redis.call('ZCARD', key)\n" +
            "\n" +
            "-- 判断是否超过限流阈值\n" +
            "return count <= limit";

    public RedisSlideWindowLimiter(RedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    /**
     * 滑动窗口限流判断
     * @param keyPrefix 限流key前缀
     * @param windowSize 窗口大小(秒)
     * @param limit 窗口内限流阈值
     * @return true: 允许请求; false: 限流
     */
    public boolean isAllowed(String keyPrefix, int windowSize, int limit) {
        String limitKey = keyPrefix + ":slide:limiter";
        String requestId = UUID.randomUUID().toString();
        long currentTimestamp = System.currentTimeMillis() / 1000;
        
        DefaultRedisScript script = new DefaultRedisScript<>();
        script.setScriptText(LUA_SCRIPT);
        script.setResultType(Boolean.class);
        
        return redisTemplate.execute(script, 
                Collections.singletonList(limitKey), 
                String.valueOf(windowSize), 
                String.valueOf(limit), 
                requestId, 
                String.valueOf(currentTimestamp));
    }
}

核心逻辑：使用Redis ZSet存储请求时间戳，动态维护时间窗口，通过Lua脚本保证操作原子性，有效避免固定窗口的流量突刺问题。

总结

固定窗口限流实现简单、资源消耗低，适合对流量控制精度要求不高的场景；滑动窗口限流通过更精细的时间粒度控制，避免了流量突刺问题，适合高并发、对稳定性要求高的核心业务。

在实际应用中，应根据业务特性、流量规模和系统承载能力选择合适的限流策略，也可以结合多种限流算法实现多层次的流量防护体系。
